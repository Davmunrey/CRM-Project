## API — Core & Infrastructure

API bootstrap and shared infrastructure for the Fastify 5 backend. index.ts is the single entrypoint: it builds the Fastify app with a custom http serverFactory, configures trustProxy and a request-correlation id generator, registers core plugins (cookie -> helmet -> cors -> jwt -> rate-limit) in a strict order, decorates `authenticate`, then registers ~45 route plugins (public first, then JWT-protected), wires Socket.io realtime with a Redis-backed db:change broadcaster, exposes a loopback/internal-key-gated /metrics endpoint, and installs global onRequest/onResponse/error/onClose hooks. Configuration is centralized in config/env.ts via a Zod schema that fail-fast exits on invalid env. db/client.ts is a single postgres.js pool (camelCase transform, tuned for PgBouncer transaction mode) and db/redis.ts is a lazy ioredis client plus helper functions for JWT denylist, session-valid-after, OAuth/SSO state, and login lockout. Middleware provides JWT authentication (with revocation + membership checks) and role-based permission guards. Types augment Fastify/JWT and stub vendor modules. The sole worker is a 60s polling sequenceRunner that advances email-sequence enrollments.

**Cross-cutting notes:**

- Plugin registration order is load-bearing: @fastify/cookie MUST precede @fastify/jwt (cookie token extraction from 'auth_token'); helmet/cors/jwt/rate-limit follow before app.decorate('authenticate') and route registration.
- Rate limiting is two-tier: global 500/min keyed per-org (rate:org:<org>) or per-IP fallback, with a stricter 20/min per-IP override on /auth; both rely on env.TRUST_PROXY to resolve req.ip from the trusted XFF suffix (spoofing defense). The store is the shared ioredis instance for multi-instance counters.
- Auth flow spans three api-core files: types/jwt.d.ts defines the payload/user claims; middleware/auth.ts verifies + checks redis denylist (jwt:deny:) and valid-after (user:valid_after:) and re-validates the users row; routes attach it via the `authenticate` decorator. RBAC (middleware/rbac.ts) runs as a preHandler AFTER authenticate and delegates to services/permissions.roleHasPermission.
- db/client.ts is the single source of the postgres.js handle (transform: postgres.camel) consumed everywhere incl. the worker; it is intentionally low max(10) for PgBouncer transaction-pooling — do not rely on session SET vars.
- Security gating: /metrics uses raw socket peer (not spoofable req.ip) for loopback check plus optional x-internal-key; setErrorHandler scrubs >=500 messages in production and emits requestId; INTERNAL_KEY gates /internal/* and the metrics key path; DEBUG_TOKEN gates /_debug/*.
- Cross-area dependencies: index.ts pulls from services/* (observability resolveRequestId/captureException, realtime registerRealtimeHandlers/broadcastDbChange, metrics registry/counters, ai/retention) and all routes/* plugins — those live outside this partition. The worker depends on services/email.sendEmail and services/tokenCipher.decryptToken.
- Realtime db:change emission is driven by ROUTE_TABLE_MAP in index.ts mapping URL prefixes to table names (e.g. /goals->sales_goals, /sequences->email_sequences, /automations->automation_rules, /orgs->organization_members, /custom-fields->custom_field_definitions); only fires on 2xx mutations with an org claim.
- env.ts fail-fast: invalid env logs fieldErrors and process.exit(1) at import time, so every module importing `env` transitively enforces config validity at startup.
- vendor.d.ts stubs (prom-client, @socket.io/redis-adapter) are placeholders replaced by real types post-install; relevant if type errors appear they may be masked by these ambient declarations.

## API Routes — Identity, Governance & Admin

Identity, governance, and admin HTTP route handlers for the Fastify 5 API. Covers local credential auth (bcrypt + JWT in HttpOnly cookies, account lockout, TOTP MFA), OIDC SSO with JIT provisioning, SCIM 2.0 IdP provisioning (Bearer api-key scoped 'scim'), org-scoped programmatic API keys + lead-capture tokens with scope enforcement, the public lead-capture API, GDPR/CCPA data-subject rights, the super-admin console (orgs/users/plans/security-events/impersonation), org self-management + member lifecycle (server-side RBAC), invitations, user preferences, Stripe billing/checkout/webhooks, plus operational health/internal/debug endpoints. Auth is primarily via app.authenticate (JWT) decorated on FastifyInstance, with header-based guards for internal (X-Internal-Key), debug (X-Debug-Token), SCIM (Bearer api_key), and public API (X-API-Key). Routes are registered under prefixes in api/src/index.ts (/auth, /auth/sso, /scim/v2, /public/v1, /orgs, /invitations, /integrations, /admin, /_debug, /preferences, /billing, /privacy, /internal). The DB client uses postgres.js with transform: postgres.camel, so snake_case columns surface as camelCase in result rows.

## API Routes — CRM Core

CRM core route plugins under api/src/routes/. Each exports an async `*Routes(app: FastifyInstance)` plugin registered with a path prefix in the server bootstrap. Every plugin authenticates via `app.authenticate` (onRequest hook or per-route `auth = { onRequest: [app.authenticate] }`). Multi-tenancy is enforced on every query by filtering `organization_id = req.user.org` (org id from JWT); writes also set it. The DB layer is postgres.js with `transform: postgres.camel`, so SQL uses snake_case columns but result rows are camelCase — handlers build dynamic UPDATEs either with the `db(updatesObject)` helper (snake_case keys) or COALESCE-per-column patterns. Newer/sensitive resources (contacts, companies, deals, activities, leads) layer in `requireCrudPermission(resource)` which derives `<resource>:<read|write|delete>` from the HTTP method; pipelines/audit use inline role checks (owner/admin/manager). Input is validated with zod safeParse returning 400 on failure; deals additionally validates cross-tenant FK ownership, and contacts/deals enforce plan limits via checkPlanLimit and fire Slack notifications.

- Dynamic UPDATE patterns: contacts/companies/deals/pipelines/views/distributionLists build a `updates` object and use postgres.js `db(updates)` helper (snake_case keys); activities/leads/products/templates/goals/customFields use COALESCE-per-column SQL. JSON columns written via db.json(): deals.quote_items, pipelines.stages, views filters; leads/customFields use JSON.stringify for tags/options/value/metadata.
- Side effects: contacts POST and deals POST call sendSlackNotification (from ./slack.js, fire-and-forget void). contacts POST + deals POST call checkPlanLimit (from ./billing.js) returning 402 when over plan.
- Validation: all bodies/queries use zod safeParse -> 400 {error:'Invalid request'} (some include details: error.flatten()). Pagination caps: contacts limit<=500; companies/deals default 100. Hardcoded LIMITs: activities 500, leads scoring all, notifications 200, audit 500, lead score-snapshots 30.
- analytics.ts deals-by-stage and forecast resolve per-deal stage probability by joining the pipelines.stages JSONB array (jsonb_array_elements LATERAL, elem->>'name' = d.stage), defaulting to 0.5 when missing. Note terminal-stage strings are inconsistent: deals.ts PATCH treats 'closed_won'/'closed_lost' as canonical, pipelines defaultStages use ids 'won'/'lost' — analytics filters on 'closed_won'/'closed_lost'.
- pipelines GET / auto-provisions a default 'Sales' pipeline (defaultStages: lead/qualified/proposal/negotiation/won/lost with probabilities) for orgs that have none. DELETE is a soft delete (is_archived=true) and refuses to archive the default pipeline.
- analytics sales-reps joins activities to users by name (a.created_by = u.name) rather than by id — fragile coupling worth noting.
- contacts.ts PATCH has a legacyMap allowing snake_case keys from legacy callers (including lead_score and linked_deals which are not in the zod contactBody schema) to pass through into the UPDATE — these bypass zod validation.

## API Routes — Integrations, Comms & Automation

Fastify 5 route handlers for n0CRM's external integrations, comms, and automation. Each module is a `register`ed plugin with an explicit URL prefix set in api/src/index.ts. Most routes are JWT-guarded via `app.authenticate` (per-route `onRequest` or a plugin-wide `addHook('onRequest')`); a small set of inbound/tracking endpoints are intentionally public. Multi-tenancy is enforced everywhere by scoping every SQL query on `organization_id` (and often `user_id`) from `req.user.org`/`req.user.sub`. Postgres is accessed through tagged-template `db` (postgres.js, transform: camel) so SQL returns camelCase columns. Secrets (refresh tokens, SMTP passwords, Slack/Zoom/webhook secrets) are stored encrypted via `encryptToken`/`decryptToken` (tokenCipher). SSRF defense is layered: Slack uses a fixed-host allow-list re-checked at send time; SMTP and outbound webhooks resolve+pin a public IP via `resolvePublicIp` (ssrfGuard) to defeat DNS-rebinding TOCTOU. Inbound webhooks (generic, Zoom) verify HMAC-SHA256 signatures; Google Calendar push and the email tracking pixel/click are unauthenticated by design.

- Mount prefixes (api/src/index.ts): /email, /email-tracking, /gmail, /calendar, /webhooks, /webhook-subscriptions, /smtp, /ux-metrics, /slack, /zoom, /automations, /sequences, /ai. Full paths = prefix + in-file path. The frontend reaches tracking pixels at env.APP_URL + '/api/email-tracking/...' so the public API base also carries an /api prefix at the edge.
- PUBLIC (no JWT) endpoints — attack surface: GET /email-tracking/open, GET /email-tracking/click (both rate-limited 60/min), POST /webhooks/inbound/:orgSlug (HMAC x-hub-signature-256), POST /zoom/webhook/:orgId (HMAC x-zm-signature), POST /calendar/webhook (NO signature verification — only trusts X-Goog-Channel-ID lookup), GET /gmail/oauth-configured.
- SSRF defenses are inconsistent by design: Slack = fixed-host regex allow-list re-checked at send (slack.ts) with dedicated tests; SMTP (smtp.ts) and outbound webhook subscriptions (webhookSubscriptions.ts) = resolvePublicIp + IP pinning (services/ssrfGuard) to prevent DNS-rebinding TOCTOU; webhookSubscriptions additionally strips dangerous custom headers and requires https://. The legacy /webhooks POST register (webhooks.ts) does NOT SSRF-validate the stored url (delivery presumably guarded elsewhere).
- Signature verification differences: webhooks.ts uses crypto.timingSafeEqual (constant-time); zoom.ts compares HMAC with plain string '===' (not constant-time) — potential timing-side-channel nuance. Generic inbound prefers req.rawBody (config.rawBody) and falls back to JSON.stringify(req.body); zoom.ts always JSON.stringify(req.body) which can differ from the exact bytes Zoom signed.
- Secrets: tokenCipher.encryptToken/decryptToken wraps Google refresh tokens (gmail_tokens.refresh_token_cipher), SMTP passwords (org_smtp_settings.password_enc), Slack webhook URL, Zoom secret (organizations.settings.*Cipher), generic webhook secret (webhooks.secret), and outbound signing secrets (webhook_subscription_secrets.signing_secret). decrypt failures generally fall back to treating the stored value as plaintext (legacy migration tolerance).
- OAuth CSRF/state binding for Gmail relies on Redis helpers storeOAuthState/consumeOAuthState (db/redis.js) keyed to req.user.sub; mismatch -> 403 at oauth-exchange. PKCE (code_challenge S256 / code_verifier) is used.
- Multi-tenant isolation pattern is uniform: every query filters organization_id from req.user.org and most also user_id from req.user.sub; cross-entity references (contact_id/company_id/deal_id/ownerUserId/sequenceId) are explicitly ownership-checked before insert in email.ts, gmail.ts, emailTracking.ts, sequences.ts.
- RBAC: only webhookSubscriptions.ts enforces fine-grained permission via requirePermission('webhooks:manage') (middleware/rbac.js). email.ts hard-codes a viewer-role block on send. Other modules rely solely on app.authenticate.
- Shared external services depended on: services/email.sendEmail, services/ai/{providers,agent,tools}, services/tokenCipher, services/ssrfGuard, db/redis. Tables touched span this area + core CRM (contacts, companies, deals, activities, notifications, users, organizations.settings).
- ai.ts governance is the spend-control choke point: per-org enabled flag (503), monthly output-token cap (429) computed from ai_usage_log, and a global rate limit of 30/min vs the app default 500/min. allowWrites gates the agent's write tools.

## API Services — AI Subsystem

The AI subsystem provides a multi-provider LLM abstraction, a provider-agnostic tool-using agent loop, an org-scoped CRM tool catalog, and a retention purger. `providers.ts` defines one normalized message/tool/usage format and translates it to Gemini, OpenAI, and Anthropic wire formats, with a host allow-list, hard fetch timeout, key-based availability, and a preference-ordered provider factory (Gemini is the free default). `agent.ts` runs an injected provider through a bounded tool-call loop (`runAgent`) plus several one-shot helpers (summarize, draft reply, next-best-action, semantic search). `tools.ts` is the CRM tool registry — every tool filters on `organization_id`, writes verify FK ownership and require `allowWrites`, and writes append an `audit_log` row. `retention.ts` periodically purges expired `ai_conversations`/`ai_messages`/`ai_usage_log` rows when `AI_MESSAGE_RETENTION_DAYS > 0`. The loop and tools are designed for unit testing with fakes (no DB/network needed for the agent loop).

**Cross-cutting notes:**

- Security model summary: encryption at rest = AES-256-GCM (tokenCipher.ts, 32-byte hex TOKEN_ENCRYPTION_KEY, iv|tag|ciphertext); session = JWT in HttpOnly/SameSite=strict cookie (cookieAuth.ts) verified by fast-jwt with env.JWT_SECRET; OIDC SSO verifies ID tokens via JWKS RS256 signature + iss/aud/exp(+60s skew)/nonce, with PKCE S256; RBAC = static role->permission Set matrix with owner=all (permissions.ts); SSRF guard blocks RFC1918/loopback/link-local/CGNAT/IPv6-ULA and pins resolved IP to defeat DNS rebinding; TOTP = RFC 6238 SHA1/6-digit/30s with constant-time compare and ±1-step window.
- Realtime auth (realtime.ts) mirrors HTTP auth middleware: JWT verify + Redis jti denylist (isTokenDenied) + tokensValidAfter session-replacement check + live DB check that the user is active and still belongs to the claimed org; without the DB check a deactivated/org-removed user would keep receiving events until JWT expiry (up to 7 days). Socket.io rooms: org:${orgId} (broadcasts) and user:${userId} (targeted); presence:join/presence:leave emitted to org room.
- Cross-area dependencies: services consume config/env.js (JWT_SECRET, OIDC_*, SMTP_*, RESEND_API_KEY, EMAIL_FROM, REDIS_URL, TOKEN_ENCRYPTION_KEY, SENTRY_DSN), db/client.js (postgres.js tagged-template `db`, transform postgres.camel), and db/redis.js (isTokenDenied, getUserTokensValidAfter). permissions.ts is enforced by middleware/rbac.ts requirePermission(); realtime debounce calls broadcastDbChange after DB writes.
- DB tables referenced: security_events (columns event_type, actor_user_id, actor_email, organization_id, ip, user_agent, detail — migration 020_security_events.sql) and users (id, organization_id, is_active) for the socket auth check.
- observability.ts and oidc.ts contain explicit Sentry/error-tracker wiring TODOs: captureException is the single hook to call Sentry.captureException when SENTRY_DSN is set, but no @sentry/node SDK is actually initialized — currently log-only.
- Gotchas: email.ts silently no-ops (console.warn) when no transport is configured rather than throwing — callers won't see send failures; resolveFrom sanitizes only the display name, not the address. assertPublicHost in ssrfGuard.ts is deprecated in favor of resolvePublicIp because awaiting validation then connecting by hostname reopens the TOCTOU rebinding window — callers must connect to the returned IP. tokenCipher.encryptToken returns null (not throw) when key missing, so callers must handle null to avoid silently storing plaintext.

## Database — Schema & Migrations

Authoritative data model for n0CRM, a multi-tenant CRM. 21 plain-SQL migration files in api/migrations/ applied in lexical order by a custom runner that records applied files in a _migrations table (referenced in 017). Note the duplicate numeric prefix 002 (002_password_reset_tokens.sql and 002_indexes_and_perf.sql) — they are distinct files.

**Cross-cutting notes:**

- Two files share the prefix 002: 002_password_reset_tokens.sql and 002_indexes_and_perf.sql — distinct migrations, applied in filename lexical order.
- Tenant isolation is dual-layer: app-layer organization_id = $1 filtering is primary; Postgres RLS (002_indexes_and_perf.sql + 018_ai.sql) is defense-in-depth. RLS depends on the API calling set_current_org(orgId) once per request to set the app.current_org_id GUC (transaction-local; comment warns to use session-scope if the pool reuses session settings). The table-owner/app role bypasses RLS, so RLS only fences non-owner roles.
- RLS coverage gap: many later tables are NOT in any RLS policy list — gmail_tokens, gmail_thread_links, gmail_thread_workspace, webhook_subscriptions(+secrets), email_tracking_* (003), api_keys, lead_capture_tokens (004), org_smtp_settings (005), ux_metric_events (006), pipelines/pipeline_members (008), calendar_events/calendar_watch_channels (009), webhooks/webhook_events (010), plans/subscriptions (014), smart_views/inbox_views/distribution_lists/user_preferences (015), impersonation_logs (016). Only the 001 core tables + the three 018 AI tables have RLS policies. These rely solely on app-layer org filtering.
- Column naming is snake_case; postgres.js transform: postgres.camel exposes them as camelCase in TS (e.g. organization_id -> organizationId, mfa_secret_cipher -> mfaSecretCipher).
- Encrypted-at-rest secrets: gmail_tokens.refresh_token_cipher, org_smtp_settings.password_enc, users.mfa_secret_cipher (AES-256-GCM via services/tokenCipher.ts), webhook_subscription_secrets.signing_secret, api_keys.key_hash / lead_capture_tokens.token_hash (hashed, never plaintext).
- Currency defaults are inconsistent across tables: organizations.currency/products.currency/plans.currency default 'EUR' while deals.currency defaults 'USD'.
- entity_id columns in notifications, audit_log, custom_field_values are plain text (not uuid FKs), so cross-entity references there are loosely typed and not FK-enforced.
- Org status appears in two places with different value sets: organizations.plan is free-form text (default 'free') while organizations.status (016) is CHECK active|suspended|trial; subscriptions.status (014) is a separate billing lifecycle.
- 002_indexes_and_perf.sql uses CREATE INDEX CONCURRENTLY which cannot run inside a transaction block — the migration runner must apply this file outside BEGIN/COMMIT (header comment documents this).
- completed tasks #25-27 mention RBAC/member-lifecycle and SCIM 2.0 provisioning, but no migration file in this partition (001-020) adds SCIM-specific tables/columns — that work appears to reuse existing users/invitations/role columns or lives outside api/migrations.
- Cascade behavior: deleting an organization CASCADE-deletes essentially all business data; user deletion mostly SET NULL on ownership/assignment columns (preserving records) but CASCADE on user-owned auth artifacts (password_reset_tokens, gmail_tokens, gmail_thread_links/workspace, calendar_*, user_preferences).

## Frontend — App Shell, Routing & State

The frontend-core is a React 18 + Vite SPA shell using react-router-dom v6 for routing and 24 Zustand stores for global state. main.tsx bootstraps theme/Sentry/trace-fetch then mounts App, which wraps everything in BrowserRouter > GmailTokenProvider > WorkspaceHostBootstrap + AppRoutes. Routing is defined inline in App.tsx (no dedicated router file and no vite-env.d.ts exist): all pages are React.lazy code-split, and protected pages are wrapped by ProtectedPage (ProtectedRoute + Layout + ErrorBoundary) with per-route Permission gating. Auth/session bootstrap is cookie-based (HttpOnly): authStore persists only the user profile + expiry, and initAuth() calls GET /auth/me on cold load to restore the session; ProtectedRoute redirects on missing auth/org and enforces workspace-host (subdomain) matching. Each domain store follows a consistent pattern: a row-mapper that tolerates camelCase or snake_case (postgres.camel transform), optimistic local writes with crypto.randomUUID() temp ids, then api.* calls with rollback on failure; some stores persist to localStorage via zustand persist. useDataInit() fan-out-fetches all stores once both currentUser and organizationId are present.

**Cross-cutting notes:**

- NO vite-env.d.ts exists anywhere in frontend, and there is NO dedicated router file — all routing is inline in App.tsx via react-router-dom v6 Routes/Route. The only related files are components/auth/ProtectedRoute.tsx (guard) and lib/routePreload.ts (not in this partition).
- Auth is HttpOnly-cookie based: tokens are NOT stored in JS. authStore persists only currentUser/session(expiresAt unix ms)/organization/organizationId/invitations. Cold-load bootstrap = App.tsx useEffect -> initAuth() which GET /auth/me; isLoadingAuth gates ProtectedRoute to avoid /login flash (AUTH-04).
- Multi-tenant + workspace-host model: ProtectedRoute redirects no-org users to /org-setup (or /org-access-required when tenantResolutionStatus==='needs_invitation', AUTH-06). WorkspaceHostBootstrap resolves a subdomain slug via GET /auth/resolve-org/:slug and sets workspaceHostMismatch, which forces a logout panel.
- Store write pattern is uniform: optimistic local mutation with crypto.randomUUID() temp id, then api.* (lib/api.ts); on POST success the temp row is swapped for the server row via row-mapper; failures roll back and often toast.error(). Most fire-and-forget PATCH/DELETE swallow errors.
- Row-mappers in every domain store accept BOTH camelCase and snake_case keys (e.g. r.createdAt ?? r.created_at) because the API uses postgres.js transform: postgres.camel but some payloads/JSON columns arrive snake_case; JSON columns (trigger, actions, steps, flowDefinition, variables, options, tags) are conditionally JSON.parse'd.
- Cross-store coupling: dealsStore.moveDeal -> notificationsStore + automationsStore/POST /automations/trigger; automationsStore.executeRulesForTrigger -> activities/notifications/deals stores; activitiesStore.addActivity -> leadsStore.recomputeLeadScore; leadsStore.convertLeadToContact -> companies/contacts stores; many stores -> auditStore.logAction and toastStore.toast.
- Persisted (localStorage) stores: authStore('crm_auth'), settingsStore(LS_KEYS.settings), attachmentsStore('crm_attachments'), distributionListsStore('crm_distribution_lists'), navigationPrefsStore, onboardingStore('crm_onboarding_v1'), viewsStore('crm_views'), emailStore('crm_emails_v2' v6 — migrate fn + partialize strips email bodies to keep storage small). Non-persisted stores fetch fresh via useDataInit.
- useDataInit (hooks/useDataInit.ts) is the central fan-out that fetches 18 stores once currentUser+organizationId exist, plus initRealtimeSubscriptions() (Socket.io) and timers (30min lead-score maintenance, 20s silent deal sync, refetch on 'online').
- settingsStore drives runtime theming: App.tsx subscribes to it and calls applyTheme/applyUiDensity/applyBrandingAccentToDocument; it is purely client-side (no API). googleClientId and emailIdentities/signatures also live here.
- AI surface (aiStore) hits /ai/status, /ai/agent (tool-using agent loop with allowWrites flag + steps), /ai/summarize, /ai/draft-reply, /ai/next-best-action — relevant to AI governance per-tenant disable noted in repo memory.
- Email tracking is security-relevant: emailStore injects an open-pixel and rewrites outbound links to /api/email-tracking/* using crypto.randomUUID() tokens; tracking-message + link rows are POSTed before send and metrics polled via GET stats. Gmail OAuth access token is held only in GmailTokenContext (in-memory, never persisted).
- getErrorMessage is imported from lib/supabaseHelpers in several stores — legacy Supabase naming remains even though backend is Fastify/postgres.js (per repo memory, Supabase legacy was being removed).

## Frontend — Pages & Feature Modules

The frontend pages layer is a React 18 + Vite SPA using React Router v6. Every page consumes Zustand stores (camelCase transforms from the Fastify API) and i18n via useTranslations(). Auth/permission gating is enforced inline via PermissionGate components and hasPermission() checks. The inbox is a full Gmail integration (OAuth2 token refresh, thread-level CRUD, tracking pixels) rendered in a three-panel layout (FolderNav / ThreadListPanel / ReadingPane). The sequences feature uses @xyflow/react for a node-based flow studio that generates SequenceFlowDefinition JSON which drives enrollment logic. Settings is a tabbed mega-page routing via ?tab= query param across 13 subsections. Pages are split into a flat top-level layer plus three subdirectories: pages/deals/ (kanban + list + quote), pages/inbox/ (folder nav + thread + reading pane), and pages/settings/ (one file per tab section).

**Cross-cutting notes:**

- All pages consume useTranslations() for i18n; camelCase field names come from postgres.js camel transform via Zustand stores, never raw SQL names.
- PermissionGate wraps every mutating action; underlying hasPermission() checks UserRole (admin|manager|sales_rep|viewer) against a Permission set.
- The Deals page uses ?create=1 and ?deal=<id> URL params as deep-link triggers (mirrored in Contacts and Activities pages).
- InboxPage drives a 15-second scheduled email processor interval and a 5-second snooze wakeup interval, both stopped on unmount.
- The sequences-flow feature depends on @xyflow/react (React Flow v12); its SequenceFlowDefinition JSON is versioned via SEQUENCE_FLOW_VERSION constant.
- QuoteBuilder in DealsPage generates PDF files client-side via jsPDF and can inject them as base64 attachments into the EmailComposer draft.
- SettingsPage routes tab state via ?tab= query param (not React Router nested routes) so the URL is shareable.
- Admin.tsx is only accessible to super-admin role; it calls /admin/* API endpoints not available to org-level users.
- CompanyDetail.tsx path not explicitly listed in glob results but exists based on navigation — confirmed by imports in Contacts.tsx (Link to /companies/:id).

## Frontend — Components

React 18 + TypeScript component layer for the n0CRM (Velo) frontend, organized by domain: activities, ai, auth, brand, companies, contacts, deals, email (+ email/composer), import, integrations, layout, settings, shared, ui (design-system primitives), and workflows. Domain forms (Contact/Company/Deal/Activity) all use react-hook-form + zodResolver with i18n-derived schemas (createXSchema(t)) and compose the ui/ primitives (Input, Select, Textarea, Button). The ui/ folder is the design system: token-based Tailwind primitives (no raw palette — uses bg-surface-*, text-fg*, accent-*, semantic success/warning/danger/info), with Select delegating to a portal-based SearchableSelect, Modal/SlideOver/ConfirmDialog sharing focus-trap logic, and a SlideOver-based global AiAssistant. All copy is internationalized via useTranslations()/getTranslations() from ../../i18n, and most stateful pieces read Zustand stores (often via manual subscribe() to avoid Zustand v5 + StrictMode getSnapshot issues, notably in Topbar/Sidebar/SmartViewBar/AttachmentsList). Integration cards and settings panels talk to the Fastify API via lib/api (camelCase responses per postgres.camel).

## Frontend — Lib, Hooks, Utils & Services

Frontend support layer for the React 18 SPA: the HTTP client, validation schemas, permission/RBAC helpers, formatting/utility functions, React hooks, and email/Google integration services. The api client (lib/api.ts) is the single fetch wrapper — cookie-based auth (HttpOnly auth_token via credentials:'include'), no Authorization header, with a structured error contract (thrown Error carries .status and .body, auto-redirects to /login on 401 unless already on /login). Validation uses Zod factory functions that take i18n Translations to localize messages. Permissions are a client-side mirror of server RBAC: a static role→Permission map (DEFAULT_ROLE_PERMISSIONS) with optional dynamic per-org overrides from the settings store. Email sending is abstracted behind an EmailProvider interface with gmail (direct OAuth/PKCE to Gmail REST), resend, and smtp (both via POST /email/send) implementations selected by VITE_EMAIL_PROVIDER. Realtime is Socket.io (db:change events) triggering throttled store re-fetches.

**Cross-cutting notes:**

- api client error contract is the load-bearing cross-cutting detail: callers (stores/services) rely on thrown Error having .status (number) and .body (parsed JSON, error message read from body.error). 401 auto-redirects to /login unless the current path already starts with /login (so the login page can surface bad-creds/MFA errors). Uses cookie auth (credentials:'include', HttpOnly auth_token) — no token in JS, no Authorization header.
- Permissions are a CLIENT-SIDE MIRROR only — getPermissionsForRole reads dynamic per-org overrides from useSettingsStore (settings.permissionProfiles) but hasPermission/hasAnyPermission/hasAllPermissions use the STATIC ROLE_PERMISSIONS map (no dynamic override). Server-side RBAC is the real enforcement (see backend partition). 'owner' role is normalized to 'admin' in authStore before reaching these helpers.
- Validation schemas are Zod factory functions taking i18n Translations (t) so error messages localize; createDealSchema additionally takes availableStages to validate against the dynamic pipeline. csvRowSchemas use plain Zod (no i18n) with safe-parse + sensible defaults for imports.
- Email sending is provider-abstracted: gmail goes DIRECT to Gmail REST from the browser using an OAuth access token (PKCE flow in gmailService), while resend/smtp route through the backend POST /email/send. Header sanitization (strip CR/LF and <>) appears in both gmailService MIME builders and outboundEmailIdentity to prevent header injection.
- Realtime: realtimeSubscriptions.ts connects Socket.io with withCredentials (cookie handshake), listens for 'db:change' {table} and throttles (180ms) store re-fetches; also exposes window.__n0crmDbChange bridge. useDataInit orchestrates the initial parallel store hydration and polling intervals.
- supabaseHelpers.ts is legacy-named but now just wraps the n0CRM api client (sbDelete/sbBulkDelete hit /<table>/<id>); confirms the Supabase->postgres.js migration. usePresence is an explicit stub returning empty members (Socket.io presence not yet wired).
- Dependencies on other partitions: types/* (Contact, Deal, Company, Activity, Lead, AppSettings, SmartViewFilter, navigation types, GmailThread/Message), i18n/* (Translations, Language, seed ids), store/* (auth, settings, and all entity stores), config/navigationDefaults, components/ui/Badge, and data/linkedin-industries-v2.json.
- Security-relevant: securePassword uses crypto.getRandomValues (min length 12, excludes ambiguous chars); gmailService OAuth uses PKCE S256 + CSRF state nonce stored in sessionStorage; useGoogleOAuthPopup validates postMessage origin === window.location.origin; emailTracking and emailPlainFormatting both HTML-escape user content before injecting into HTML bodies.

## Frontend i18n, Infra, Ops, Config & Docs

The i18n subsystem is a fully-typed, Zustand-persisted catalog. `types.ts` defines the `Translations` interface (the single source of truth for every UI string) and `WorkflowLibraryCatalog` for automation/sequence seed copy. `en.ts` is the canonical full implementation; `es.ts` is also a full implementation with no spread inheritance. `pt.ts`, `fr.ts`, `de.ts`, and `it.ts` all spread `...en` and override only localised sections, so they are "inherit-from-en" locales. The `workflowLibrary/` subtree provides per-locale seed copy for automations and sequences. The coverage test in `tests/i18n/coverage.test.ts` diffs every string-leaf path in all locales against `en` and fails on any missing key. The infra layer is a Docker Compose monolith: Postgres 16 + PgBouncer (transaction pooling, port 6432) + Redis 7 + the Fastify API + nginx/frontend + Prometheus + Grafana + a cron backup container. `infra/` holds the static config files for PgBouncer, Prometheus, Grafana auto-provisioning, and the pg_dump/cron backup scripts. Docs are split between the repo root (README, SECURITY-AUDIT) and `docs/` (SSO/SCIM ops guide, disaster-recovery runbook, ADR 0001 on tenant isolation).

  - Redis: ephemeral by design — not backed up (JWT denylist, rate-limit counters recreatable)
  - Recovery objectives table; off-site copy procedures
  - Quarterly restore drill instructions (release blocker closed by this doc)
- **[`docs/adr/0001-tenant-isolation-and-rls.md`](docs/adr/0001-tenant-isolation-and-rls.md)** — Architecture Decision Record — accepted policy on tenant isolation and RLS status
  - Status: Accepted, Date: 2026-06-11
  - Application-layer org scoping (WHERE organization_id = req.user.org) is the primary control
  - RLS policies on ~21 tables are dormant: API connects as table-owner (bypasses RLS); PgBouncer transaction mode issues DISCARD ALL so session GUC unreliable
  - Decision: do NOT call set_current_org() per request; keep app-layer as authoritative; document RLS as dormant defense-in-depth
  - Risks and migration path to activate RLS in future documented
